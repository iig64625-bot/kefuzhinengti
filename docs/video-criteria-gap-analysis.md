# 合格 Agent 项目标准对照（基于《懂点AI的老李》视频）

> 视频：什么样的Agent项目才有含金量？能真正给你的简历加分？  
> 作者：懂点AI的老李  
> 链接：https://www.douyin.com/video/7655546775524183331

## 一、视频核心论点（6 条标准）

| # | 标准 | 一句话 |
|---|------|--------|
| 1 | **真实业务场景** | 有明确用户、完整业务流程、可量化效果 |
| 2 | **完整后端工程** | API、数据库、会话、日志，不是平台拖拽 Demo |
| 3 | **Agent 核心能力** | 规划、工具调用、失败恢复、结果验证 |
| 4 | **上下文工程** | 什么信息何时进 Prompt（RAG / 记忆 / 状态） |
| 5 | **可观测 + 评估** | Trace 可回放 + Eval 集可量化 |
| 6 | **人工干预** | 高风险场景 Human-in-the-loop |

总括：**有价值的 Agent = 围绕业务构建的智能应用系统，不是 Chatbot 皮肤。**

## 三、升级路线（按视频标准补齐）

### Phase A：最小「含金量」闭环（3～5 天）— 必做

1. **落地 Human-in-the-loop**  
   - 检索为空 / 投诉词 / 订单核实 → IF 分支转人工模板  
   - README 写清：哪些场景必须人工确认

2. **补齐 Eval + Trace**  
   - 跑完 `tests/golden-questions.md`，填 `docs/eval-report.md`  
   - 每次运行记录：问题 → 检索片段 → 工具结果 → 最终回答 → 是否兜底

3. **做一个真实工具（Mock API 也行）**  
   - `query_order(order_id)` 返回 JSON  
   - Agent 在订单问题优先调用工具，失败才说转人工

4. **简历叙事模板（按视频逻辑重写）**  
   - 不是「用了 Dify 做客服」  
   - 而是：「针对电商助眠枕售后咨询成本高、多跳 FAQ 体验差，搭建 RAG+工具调度 Agent；20 题离线评测准确率 X%；高风险场景强制转人工」

### Phase B：工程系统感（1 周）— 加分

5. **薄后端层**（哪怕 FastAPI 百行）  
   - `/chat` 转发 Dify API  
   - PostgreSQL/SQLite：`conversations` / `messages` / `tickets`  
   - 会话 ID、用户 ID 持久化

6. **失败恢复**  
   - 工具超时重试 1 次  
   - Schema 校验失败 → 降级为 FAQ 回答  
   - 连续失败 → 转人工并写工单

7. **上下文工程写清楚**  
   - System（永久规则）  
   - Context（本轮 RAG）  
   - Memory（近 5 轮摘要）  
   - Tool Result（订单工具输出）  
   - 各自何时注入、何时截断

### Phase C：高分项（可选）

8. Multi-Agent 轻量版：Planner（意图）+ Worker（检索/工具）+ Critic（答案校验）  
9. 成本面板：单次对话 token / 费用估算  
10. A/B：Workflow 四分支 vs Agent 单链路对比报告

## 四、面试 1 分钟讲法（按视频框架）

> 我做的是**电商客服业务 Agent**，不是调库 Demo。  
> 用户进线后，系统做意图分流与知识召回；订单类走 **工具查询**；答不出或高风险走 **人工确认**。  
> 有 **20 题离线评测**和 **run 级日志回放**，可量化准确率和兜底率。  
> 难点是：模型会幻觉订单状态——我用 **工具结果校验 + 人工升级** 解决。
