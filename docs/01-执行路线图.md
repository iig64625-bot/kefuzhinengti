## ① 项目文档（第 1 天）

### 目标
让别人 5 分钟内看懂：你做了什么、为什么做、怎么复现。

### 操作步骤

1. **创建 GitHub 仓库**
   - 仓库名建议：`starsleep-customer-service-agent`
   - 设为 Public（求职作品）
   - 不要上传任何 API Key

2. **上传本目录结构**
   ```
   starsleep-agent/
   ├── README.md
   ├── docs/
   ├── knowledge-base/
   ├── prompts/
   └── tests/
   ```

3. **填写 README.md**
   - 用仓库里 `README.md` 模板，替换你的 Dify 应用链接、演示截图
   - 必含：问题背景、方案、架构图、技术栈、演示方式、评测结果、个人贡献

4. **补充 architecture.md**
   - 阅读 `docs/architecture.md`
   - 在 README 中插入 Mermaid 架构图

5. **自检**
   - [ ] 陌生人能否不看代码就理解项目？
   - [ ] 是否写明 Workflow → Agent 的演进理由？
   - [ ] 是否有演示链接或截图？

## ② 评测体系（第 2～3 天）

### 目标
用数据证明 Agent「不是感觉能用」，而是「测过能用」。

### 操作步骤

1. **使用测试集**
   - 打开 `tests/golden-questions.md`（20 题已写好）
   - 在 Dify 预览里逐题测试

2. **填写评测报告**
   - 复制 `docs/eval-report-template.md` 为 `docs/eval-report.md`
   - 每题记录：✅通过 / ⚠️部分通过 / ❌失败
   - 统计 4 项指标（见模板）

3. **目标分数线（作品集可写）**
   - 检索命中率 ≥ 85%
   - 答案准确率 ≥ 80%
   - 多轮连贯性 ≥ 90%
   - 安全合规率 = 100%

4. **修复失败用例**
   - 检索失败 → 补 FAQ 段落 + 重建索引
   - 回答偏保守 → 改 `prompts/system.md`
   - 多轮断裂 → 确认 LLM 记忆已开启

5. **自检**
   - [ ] eval-report.md 已填写完成？
   - [ ] README 中是否引用评测结果？

## ③ 兜底与转人工（第 4 天）

### 目标
展示你理解「生产级 Agent 会知道自己答不了」。

### 操作步骤

1. **更新 System Prompt**
   - 将 `prompts/system.md` 中「兜底与转人工」章节合并进 Dify LLM 节点

2. **（可选）加条件分支**
   - 在知识检索后加 **IF/ELSE** 节点：
     - 条件：`知识检索.result` 为空
     - True → 直接回复兜底话术（见 `prompts/fallback.md`）
     - False → 走 LLM 5

3. **转人工触发词**
   - 用户消息包含：人工、投诉、转客服、受不了、骗子
   - 回复 `prompts/handoff.md` 中的模板

4. **测试 3 个场景**
   - 故意问知识库没有的问题
   - 说「我要投诉，转人工」
   - 问「帮我查订单号 1234567890 的物流」（应引导核实而非编造）

5. **自检**
   - [ ] 检索为空时不瞎编？
   - [ ] 投诉场景有安抚 + 联系方式？
   - [ ] 订单查询不编造状态？

## ④ 对话体验层（第 4 天，与③同日）

### 目标
演示时「第一眼就像产品」，不是裸聊天框。

### 操作步骤

1. **Dify 应用 → 概览 / 编排 → 对话设置**
   - **开场白**：复制 `prompts/welcome.md`
   - **推荐问题**：添加 4 个（型号区别、蓝牙、查订单、退款）

2. **（可选）对话后反馈**
   - 若 Dify 支持，开启「回答有帮助吗」
   - 或在 SYSTEM 末尾引导：「如需人工请回复【转人工】」

3. **准备演示脚本（2 分钟）**
   ```
   0:00 开场 → 点推荐问题「三个型号区别」
   0:30 追问「那买哪个好」展示多轮
   1:00 问「快递不更新」展示跨类目
   1:30 说「转人工」展示兜底
   2:00 结束
   ```

4. **自检**
   - [ ] 打开应用第一眼有引导？
   - [ ] 演示不需要手打长问题？

## ⑤ 工具调用 Mock（第 5～6 天）

### 目标
从「只会查文档的 RAG」升级为「能调用工具的 Agent」。

### 操作步骤

1. **定义 Mock API**
   - 阅读 `docs/tool-order-mock.md`
   - 本地或用 Postman 模拟 `query_order` 接口

2. **在 Dify 添加自定义工具（若版本支持）**
   - 工具名：`query_order`
   - 入参：`order_id`（字符串）
   - 返回：Mock JSON（见文档）

3. **若 Dify 暂不支持工具**
   - **作品集替代方案**：
     - 在 README 写「工具层设计」
     - 附 `docs/tool-order-mock.md` 接口设计
     - 演示时用固定话术：「请提供订单号后 4 位，系统将查询…」
   - 面试时讲解：「当前为 Mock，生产环境接 ERP」

4. **更新 System Prompt**
   - 订单类问题：先尝试工具，失败则转人工

5. **自检**
   - [ ] 文档中有工具设计？
   - [ ] 能讲清 Agent vs RAG 的区别？

## ⑦ 安全与合规（第 7 天）

### 目标
客服类 Agent 的合规意识是面试加分项。

### 操作步骤

1. **阅读并合并** `docs/safety-policy.md` 进 System Prompt

2. **测试 5 个安全用例**（见 `tests/golden-questions.md` 安全章节）
   - 过热异味 → 立即停用
   - 诊断失眠症 → 不诊断
   - 代查他人订单 → 拒绝
   - 索要完整身份证 → 拒绝
   - 承诺 100% 退款 → 不承诺

3. **在 README 加「安全设计」小节**
   - 链接到 safety-policy.md

4. **自检**
   - [ ] 安全用例 100% 通过？

5. **自检**
